# modern-di-flask 3.0.1 — fix the documented root-open order

Patch release. **No public API change** — `setup_di`, `fetch_di_container`,
`FromDI`, `inject`, and `flask_request_provider` keep their signatures and
behavior. Docs-only fix plus a regression test; no adapter code changed.

## Fix

- **The README and architecture docs told the caller to open the root
  container before `setup_di`; the correct order is after.** `setup_di` is
  what registers `flask_request_provider` (`container.add_providers`), and
  modern-di 3.x defers `validate=True`'s checks to `open()` specifically so
  an integration can register its context providers first. Opening before
  `setup_di` runs validation before the `flask.Request` provider exists, so
  any provider with a non-optional `flask.Request` dependency raised
  `ValidationFailedError` at startup when following the old docs. Added a
  regression test with such a dependency, asserting the correct order
  (`setup_di(app, container)` then `container.open()`) resolves it
  end-to-end, and that the reverse order fails validation.

## Downstream

If your provider graph has a provider depending directly on `flask.Request`
(not `flask.Request | None`), open the root container **after** calling
`setup_di`, not before:

```python
container = Container(groups=[Dependencies], validate=True)
setup_di(app, container)
container.open()
```

No action needed if you already open after `setup_di`, or if none of your
providers depend on `flask.Request` directly.
