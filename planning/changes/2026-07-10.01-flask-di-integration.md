# Design: modern-di integration for Flask

**Status:** design, pending maintainer approval
**Target:** new repo `modern-python/modern-di-flask`, package `modern_di_flask`

## Summary

Ship the official [modern-di](https://modern-di.modern-python.org) integration
for [Flask](https://flask.palletsprojects.com), following the core repo's
[Writing an integration](../../../modern-di/docs/integrations/writing-integrations.md)
contract along the **middleware + `@inject` decorator** path â€” the **synchronous
twin** of the shipped `modern-di-starlette`/`-aiohttp` integrations. One module
(`modern_di_flask/main.py`) exposes `setup_di`, `fetch_di_container`, `FromDI`,
`inject`, and `flask_request_provider`. A per-request `Scope.REQUEST` child
container is built in `before_request` and closed in `teardown_appcontext`;
`@inject` (or `auto_inject=True`) resolves `FromDI` params from it.

## Motivation

Flask is one of the largest WSGI web frameworks; its synchronous model is a
clean fit for modern-di's sync resolution. It's the roadmap's next "Next up"
integration. Dishka has a Flask integration; modern-di has none.

## Verified facts (Flask 3.1.3 + modern-di 2.27, spiked end-to-end via test_client)

Every non-obvious decision below was proven against installed Flask, not inferred:

- **`before_request` / `g` / `teardown_appcontext` flow works.** `before_request`
  builds the `REQUEST` child (seeded with the current `flask.Request`) and stashes
  it on `flask.g`; `teardown_appcontext` closes it. Verified a request resolves
  app- and request-scoped values plus `request.path` (context injection), and the
  child's finalizer ran after the request.
- **No signature rewrite is needed** â€” the simplest resolution path of all the
  integrations. Flask dispatches `view_functions[endpoint](**url_args)` without
  inspecting the view signature, so the `inject` wrapper just accepts `*args,
  **kwargs`, resolves the DI params, and forwards. **`functools.wraps` is safe**
  here (it preserves `__name__` for the endpoint and Flask never unwraps to
  defeat it â€” unlike aiogram).
- **`auto_inject` wraps `view_functions` at setup time**, across the app and its
  blueprints. Verified both an app route and a blueprint route resolve without a
  per-view `@inject`, and a no-`FromDI` view is an untouched no-op. **Constraint:**
  routes/blueprints must be registered **before** `setup_di(..., auto_inject=True)`
  (same class as aiogram's "register before startup") â€” documented.
- **Flask has no app startup/shutdown hook** (WSGI). Dishka's Flask integration
  confirms this: it closes only the per-request child (in `teardown_appcontext`)
  and never the root â€” the user owns root teardown. This integration does the same.
- **`app.extensions`** is the root-container store; **`app.test_client()`** drives
  requests in-process for tests.

## Design

### Connection provider (kind â†’ scope)

One connection kind (an HTTP request â†’ `REQUEST`); Flask core has no WebSocket:

```python
flask_request_provider = providers.ContextProvider(flask.Request, scope=Scope.REQUEST)
_CONNECTION_PROVIDERS = (flask_request_provider,)
```

### Container handoff (two named keys)

- `_ROOT_CONTAINER_KEY` â€” the root container, stored in `app.extensions`, read by
  `fetch_di_container`.
- `_CHILD_CONTAINER_ATTR` â€” the per-request child, stashed on `flask.g` by
  `before_request`, read by `inject`/closed by `teardown_appcontext`.

### `setup_di(app, container, *, auto_inject=False) -> Container`

```python
def setup_di(app, container, *, auto_inject=False):
    app.extensions[_ROOT_CONTAINER_KEY] = container
    container.add_providers(*_CONNECTION_PROVIDERS)
    app.before_request(_enter_request(container))
    app.teardown_appcontext(_close_request)
    if auto_inject:
        _inject_views(app)
    return container
```

- `_enter_request(container)` returns a zero-arg callback that builds the
  `REQUEST` child seeded with `{flask.Request: flask.request._get_current_object()}`
  and sets it on `g` under `_CHILD_CONTAINER_ATTR`.
- `_close_request(_exc)` reads the child off `g` (via `getattr`, since
  `teardown_appcontext` also fires for non-request app contexts where no child was
  set) and `close_sync()`s it.
- **No root lifecycle wiring** â€” Flask has no shutdown hook. The root is created
  open and lives for the app's lifetime; APP-scoped finalizers are the user's
  responsibility (`fetch_di_container(app).close_sync()` at their own shutdown,
  or use the container as a context manager). Documented in `architecture/`.

### `fetch_di_container(app) -> Container`

```python
def fetch_di_container(app):
    return typing.cast(Container, app.extensions[_ROOT_CONTAINER_KEY])
```

### `FromDI` marker + `inject` decorator

```python
@dataclasses.dataclass(slots=True, frozen=True)
class _FromDI(typing.Generic[T_co]):
    dependency: providers.AbstractProvider[T_co] | type[T_co]

def FromDI(dependency, /):  # noqa: N802
    return typing.cast(T, _FromDI(dependency))

def inject(func):
    di_params = _parse_inject_params(func)
    if not di_params:                       # nothing to inject
        func.__modern_di_injected__ = True
        return func

    @functools.wraps(func)                  # safe for Flask; preserves the endpoint name
    def wrapper(*args, **kwargs):
        child = getattr(g, _CHILD_CONTAINER_ATTR)
        resolved = {n: child.resolve_dependency(m.dependency) for n, m in di_params.items()}
        return func(*args, **kwargs, **resolved)

    wrapper.__modern_di_injected__ = True
    return wrapper
```

`resolve_dependency` dispatches provider-vs-type. The parse/short-circuit/marker
pattern mirrors `modern-di-aiohttp`/`-aiogram`; there is **no** signature rewrite
(Flask does not need it).

### `auto_inject` (wrap views at setup)

```python
def _inject_views(app):
    for scaffold in (app, *app.blueprints.values()):
        for endpoint, view in list(scaffold.view_functions.items()):
            if not getattr(view, "__modern_di_injected__", False):
                scaffold.view_functions[endpoint] = inject(view)
```

### Public API (`__all__`)

`setup_di`, `fetch_di_container`, `FromDI`, `inject`, `flask_request_provider`.

## Non-goals

- **WebSocket** â€” Flask core has none (extensions add it); one connection kind only.
- **Async resolution** â€” sync only (`close_sync`); Flask (WSGI) is synchronous.
- **Root lifecycle wiring** â€” Flask has no app startup/shutdown hook; the user
  owns root teardown (matches Dishka).
- **A signature-rewriting `inject`** (Celery/aiogram needed one; Flask does not).
- Any runtime dependency beyond `flask` and `modern-di`; a separate docs site.

## Testing

`app.test_client()`, 100%-coverage gate (`just test-ci`), mirroring the sibling
suites:

- `dependencies.py` â€” a `Group` with app- and request-scoped `Factory` providers
  plus one reading the request (`request: flask.Request | None = None` â†’ e.g.
  `request.path`), proving context injection.
- `test_inject.py` â€” a `@inject` view resolves app-scoped, request-scoped, and
  request-derived values; a no-`FromDI` view is a passthrough.
- `test_auto_inject.py` â€” an app route AND a blueprint route resolve via
  `setup_di(..., auto_inject=True)` with no per-view `@inject` (routes registered
  before `setup_di`).
- `test_teardown.py` â€” the per-request child is closed (a REQUEST provider
  finalizer runs) after a normal request and after a view that raises (Flask's
  `teardown_appcontext` always runs); `fetch_di_container` returns the same root.

## Risk

- **R1 â€” Flask API drift.** `app.extensions`, `before_request`,
  `teardown_appcontext`, `flask.g`, `view_functions`, `blueprints`, and
  `request._get_current_object()` are relied on â€” all long-stable. *Mitigation:*
  pin `flask>=2,<4` (ruled by maintainer). The implementation step installs the
  low bound (flask 2.0) on Python 3.10 and runs the suite to confirm those APIs
  exist there; if one landed later, raise the floor to the lowest green version
  and report it.
- **R2 â€” `auto_inject` ordering.** Views registered *after* `setup_di(...,
  auto_inject=True)` are not wrapped. Documented (register routes/blueprints
  first). Explicit `@inject` has no such constraint.
- **R3 â€” `teardown_appcontext` fires outside requests.** App contexts pushed
  without a request (CLI, `app.app_context()`) trigger the teardown with no child
  on `g`; the `getattr` guard makes that a no-op.
