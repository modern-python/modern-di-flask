# modern-di-flask 3.0.0 — modern-di 3.x

Requires **modern-di >= 3, < 4**. **No public API change** — `setup_di`,
`fetch_di_container`, `FromDI`, `inject`, and `flask_request_provider` keep
their signatures. This release includes a real adapter bug fix required by
[modern-di 3.0's mandatory-open lifecycle](https://modern-di.modern-python.org/migration/to-3.x/):
the per-request child container was never opened.

## Fix

- **The per-request `REQUEST` child container is now opened.**
  `_enter_request` (the `before_request` hook) builds the child and a
  separate `teardown_appcontext` hook closes it later — two different Flask
  hooks, with no enclosing `with` block to open the child implicitly. Under
  modern-di 3.0's mandatory-open lifecycle this meant every resolve against
  the per-request child raised `ContainerClosedError`. `_enter_request` now
  calls `child.open()` right after building it, before stashing it on
  `flask.g`.

## Packaging

- **Bumped the `modern-di` dependency to `>=3,<4`.** Requires modern-di 3.x;
  drops support for modern-di 2.x.

## Downstream

**You must now open the root container yourself before serving traffic.**
Flask has no application-startup hook that `setup_di` can attach to, so
`setup_di` does not open the root container it's given — the same way root
teardown has always been the caller's responsibility
(`fetch_di_container(app).close_sync()`), opening now is too:

```python
container = Container(groups=[Dependencies], validate=True)
container.open()  # or: with container: ... — required under modern-di 3.x
setup_di(app, container)
```

Passing an unopened container to `setup_di` will raise `ContainerClosedError`
on the very first request. If your own code builds child containers
elsewhere, open them with `with` (or `open()`) before resolving — see the
[modern-di 3.x migration guide](https://modern-di.modern-python.org/migration/to-3.x/).

## Internals

- 100% line coverage; `ruff`, `ty` clean.
