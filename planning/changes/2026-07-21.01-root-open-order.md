# Change: Open the root container after `setup_di`, not before

**Lane:** lightweight — docs + one test file, no new file, no public-API
change, two straightforward tests.

## Goal

Correct a documentation/ordering defect: the README and
`architecture/dependency-injection.md` told the caller to open the root
container before calling `setup_di`. But `setup_di` is what registers
`flask_request_provider` (via `container.add_providers`), and modern-di 3.x
defers `validate=True`'s checks to `open()` specifically so an integration
can register its context providers first. Opening before `setup_di` runs
validation before the `flask.Request` provider exists, so any provider with
a non-optional `flask.Request` dependency raises `ValidationFailedError` at
startup. The 3.0.0 suite never caught this because its one request-scoped
factory (`tests/dependencies.py::fetch_request_path`) types the request
param `flask.Request | None = None`, so `validate=True` never sees a hard
dependency on it.

## Approach

- `README.md` — swap the usage example's order: `setup_di(app, container)`
  then `container.open()`; the surrounding prose now says `.open()` runs
  after `setup_di`, not before.
- `architecture/dependency-injection.md` ("No root lifecycle wiring") —
  correct the sentence that told the caller to open before `setup_di`, and
  explain why the order matters (registration vs. validation). The
  "root lifecycle is caller-owned" contract itself is unchanged.
- `tests/test_middleware.py` — add a `Group` with a `Factory` depending on a
  non-optional `flask.Request` (no default): one test asserts
  `setup_di(app, root); root.open()` succeeds and resolves end-to-end
  through `@inject`/`FromDI`; a second documents that opening before
  `setup_di` raises `ValidationFailedError`.
- No change to `modern_di_flask/main.py` — `setup_di` still does not open
  the root container, per the maintainer's existing ruling that root
  lifecycle is caller-owned.

See [architecture/dependency-injection.md](../../architecture/dependency-injection.md)
for the promoted capability description.

## Files

- `README.md` — usage example and prose: open the root container after
  `setup_di`.
- `architecture/dependency-injection.md` — "No root lifecycle wiring": open
  must come after `setup_di`, with the registration/validation rationale.
- `tests/test_middleware.py` — two new tests: hard `flask.Request` dependency
  resolves under the correct order; the reverse order fails validation.
